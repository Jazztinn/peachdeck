# Peachdeck

Create pitch decks that win on clarity, evidence, and disciplined visual storytelling. Optimize for a skeptical investor, judge, or buyer who needs to understand the company fast.

## Workflow

1. Establish the deck context.
   - Identify audience, stage, desired output, presentation length, and whether the user wants a file or slide copy only.
   - If critical inputs are missing, ask up to five concise questions. If momentum matters, proceed with explicit assumptions instead.
   - Minimum inputs: company or project, customer, problem, solution, proof or traction, team, and ask.

2. Write the core sentence before drafting slides.
   - Produce a one-sentence company description in plain language.
   - Use this shape: "For [customer], [company] [does X] so they can [outcome]."
   - Rewrite until a non-expert can repeat it after one read.

3. Choose the deck spine.
   - Read `references/deck-structure.md` for seed, hackathon, demo-day, and review-specific slide sequences.
   - Keep the main deck short: normally 6-8 slides for hackathons and 10-12 slides for seed fundraising.
   - Move dense material, technical depth, and extra charts to an appendix.

4. Draft slides as claims, not labels.
   - Make each title a conclusion, for example "Clinics lose 8 hours a week to intake rework" instead of "Problem."
   - For each slide, include: headline claim, evidence, visual direction, and optional speaker note.
   - Prefer real numbers, named customers, product screenshots, demos, charts, or specific user stories over broad claims.
   - Mark invented or inferred content as `[Assumption]`.

5. Raise the quality bar.
   - Read `references/investor-quality-bar.md` before finalizing narrative, evidence, and fundraising logic.
   - Read `references/video-notes.md` for seed-stage, pre-product-market-fit, or Michael Seibel/Y Combinator style fundraising decks.
   - Cut jargon, vanity language, vague TAM claims, and unearned adjectives.

6. Design or package the output.
   - If the user asks for a deck file, create a PPTX or other requested format with the best available presentation tooling.
   - If no file is requested, return a deck spec table with slide number, title, objective, body copy, visual, and speaker note.
   - If creating a visual deck, read `references/design-qa.md`, render or preview the slides when the environment supports it, and fix visible layout issues before delivery.

## Output Standards

- Start with what the company does, who it serves, and why the problem matters.
- Put the strongest evidence early enough that the audience does not need faith to keep listening.
- Treat team, traction, and market size as investor filter slides for seed-stage decks.
- For seed decks, explain the company in two sentences plus a concrete example before moving into proof.
- Make the ask specific: money, prize, pilot, intro, partnership, next meeting, or hiring need.
- Use direct language. Do not make the deck sound like a press release.
- Prefer fewer, stronger slides over a complete but forgettable survey of the business.

## Review Mode

When asked to review an existing deck, lead with the highest-risk issues:

1. Unclear company description or weak first minute.
2. Missing customer, problem, traction, market, team, or ask.
3. Unsupported claims or numbers that do not survive investor scrutiny.
4. Slide density, illegible charts, or design choices that hide the story.
5. Suggested replacement slide order and rewritten headlines.

End with a prioritized fix list, not a generic scorecard.
